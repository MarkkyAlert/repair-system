# UAT: Report Face-Validity Checklist (ตรวจว่ารายงาน "ตรงกับความจริงหน้างาน")

## ทำไมต้องมีขั้นนี้

Reconciliation tests (`tests/cases/*_report_test.php`, 286 เทสต์) พิสูจน์แล้วว่ารายงาน **คำนวณถูกตามนิยาม + ดึงข้อมูลตรงกับ DB + จัดการ edge case ถูก** (สูตร, หารศูนย์, นับซ้ำ, ขอบวันที่, empty≠zero, ฐานตัวอย่าง, export=จอ). สิ่งที่เทสต์ **พิสูจน์แทนคนไม่ได้** คือ **face validity** — ผู้บริหาร/หัวหน้าที่รู้ความจริงหน้างานดูแล้วรู้สึกว่า "ตัวเลขสะท้อนงานจริงที่เขาสัมผัส" ไหม

> เลขถูกทางคณิตศาสตร์ ≠ เลขสะท้อนความจริง. UAT นี้ตรวจอย่างหลัง.

## วิธีทำ (30–45 นาที)

1. **เลือกผู้ตรวจ** — หัวหน้า/ผู้จัดการที่รู้ว่าเดือน/ไตรมาสที่ผ่านมาเป็นยังไงจริง (ใครทำงานเยอะ พื้นที่ไหนมีปัญหา ทรัพย์สินตัวไหนเสียบ่อย)
2. **เลือกช่วงเวลาที่ผู้ตรวจ "รู้ดี"** — เช่นเดือนที่แล้ว แล้วตั้ง filter ทุกรายงานให้ตรงช่วงนั้น
3. **ต่อรายงาน:** เทียบ "ตัวเลขพาดหัว" กับ (ก) ความรู้ของผู้ตรวจ และถ้าเป็นไปได้ (ข) แหล่งอิสระ (นับมือ, ระบบเดิม, สมุดงาน) — ไม่ต้องเช็คทุกเลข เอาแค่ 2–3 ตัวที่ผู้ตรวจรู้คำตอบอยู่แล้ว
4. **เจอไม่ตรง → บันทึกเป็น issue** (ระบุรายงาน + ตัวเลข + ค่าที่ควรเป็น + วิธีที่ผู้ตรวจรู้) ส่งให้ทีม dev สืบว่าเป็น "นิยามต่างกัน" หรือ "ข้อมูลต้นทางผิด"

## Checklist ต่อรายงาน

| รายงาน | ตัวเลขพาดหัวที่เทียบ | เทียบกับอะไร | ธงแดง (ถ้าเจอต้องสืบ) |
|---|---|---|---|
| **สรุปผู้บริหาร** (executive) | แจ้งซ่อมทั้งหมด, ปิดงาน, อัตราปิดงาน, คะแนนเฉลี่ย (+ "จาก N รีวิว") | ความรู้สึกรวมของเดือนนั้น + จำนวนงานที่ทีมรู้ว่าปิดจริง | ปิดงานต่างจากที่ทีมรู้ ; คะแนน 5.0 จากรีวิวน้อยมากแต่ดูเหมือนดีเลิศ ; delta ทิศผิด (ดีขึ้น/แย่ลง สวนความจริง) |
| **ผลงานทีมช่าง** (technician-performance) | อันดับช่าง, อัตราปิดงาน, SLA ตรงเวลา (+จำนวนงาน), คะแนน (+จำนวนรีวิว) | ใครที่หัวหน้ารู้ว่าทำงานเยอะ/ดีจริง | อันดับสวนความจริง ; "100% / 5.0" ที่มาจากฐาน 1–2 งาน (ดูตัวเลขในวงเล็บ) ถูกใช้ตัดสินคน |
| **SLA เกิน** (sla-breach) | พื้นที่/หมวด/แผนกที่เกิน SLA มากสุด | พื้นที่/หมวดที่รู้ว่ามีปัญหาจริง | บอตเทิลเน็กที่ report ชี้ ≠ ที่หน้างานรู้ ; สองสถานที่ชื่อคล้ายกันสับสน |
| **พื้นที่ปัญหา** (problem-hotspot) | พื้นที่ที่คะแนนปัญหาสูงสุด | พื้นที่ที่ร้องเรียน/งานเข้าเยอะจริง | พื้นที่ปัญหาจริงไม่ขึ้นบนสุด |
| **แนวโน้ม** (trend) | เดือนที่งานพุ่ง / SLA ตก / คะแนนดิ่ง | เหตุการณ์ที่จำได้ (เดือนที่ยุ่ง, ช่วงมีปัญหา) | กราฟไม่สะท้อนเดือนวิกฤตที่รู้ว่ามีจริง (ระวัง 0% จริงที่เคยถูกซ่อน — แก้แล้ว แต่ยืนยัน) |
| **งานค้างตามอายุ** (backlog-aging) | จำนวนงานค้าง >30 วัน, งานเก่าสุด | งานค้างเก่าที่หัวหน้ารู้ว่ามีจริง | ตัวเลขค้างต่ำกว่าจริง ; งานเก่าสุดไม่ตรง |
| **งานเปิดซ้ำ** (reopen-rate) | หมวด/ช่างที่ %เปิดซ้ำสูง | งานที่รู้ว่า "ปิดแล้วกลับมา" บ่อย | %เปิดซ้ำ >100% (เป็นไปไม่ได้) ; หมวดที่ปัญหาจริงไม่ขึ้น |
| **ความพึงพอใจ** (csat) | คะแนนเฉลี่ย (+จำนวนรีวิว), %ไม่พอใจ | feedback ที่ได้ยินจากผู้ใช้จริง | คะแนนสูงสวนเสียงบ่นจริง ; avg สูงจากรีวิวน้อย |
| **สุขภาพทรัพย์สิน** (asset-reliability) | ทรัพย์สินที่ "ควรเปลี่ยน" / เสียบ่อย / MTBF สั้น | ตัวที่ช่างบ่นว่าเสียบ่อยจริง | ตัวที่ควรเปลี่ยนจริงไม่ขึ้น ; อายุ/MTBF ดูผิดธรรมชาติ |

## ตรวจ Export ด้วย (จอ = ไฟล์)

เปิดรายงานบนจอ → กด export **CSV / Excel / PDF** ช่วงเดียวกัน → ยืนยันตัวเลขพาดหัว **ตรงกันทุก format** (ระบบมีเทสต์ล็อกส่วนนี้แล้ว แต่ UAT ยืนยันด้วยตาอีกชั้น). ลอง export ตอน **ไม่มีข้อมูล** (ช่วงว่าง) → ต้องได้ไฟล์หัวตารางเปล่า ไม่ error และไม่มีเลขปลอม
